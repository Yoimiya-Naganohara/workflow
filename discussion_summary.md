## 1. 代码审查（初始任务）

对 Rust 项目进行了全面代码审查，涵盖：
- **正确性** — 发现 46 个问题（后来扩展到 53 个）
- **安全性** — API key 混淆、shell 注入、路径遍历
- **稳定性** — 并发安全、数值鲁棒性、panic 防护、资源管理

### 关键发现（P0 级）
| ID | 问题 | 严重性 |
|----|------|--------|
| 1 | mmap 映射文件越界：`bounds_check = ptr.add(len)` 在 len=0 时 UB | P0 |
| 2 | L2 LLM 响应解析：`[level, time, msg] = parts` 未检查长度直接索引 | P0 |
| 3 | Health tracking：`healthy_count` 非原子递减导致并发损坏 | P0 |
| 4 | Missing merge：`snapshot_files()` 合并后需 `insert_file_metadata` | P0 |

### 网络稳定性覆盖不足的自我发现
用户指出后，补充了网络稳定性审查，新增 7 个问题：
- `timeout_secs`/`max_retries`/`max_connections` 三个配置字段全部 **unused**
- `complete_ext!` 宏无 timeout，provider 挂起永久阻塞 tokio
- 无重试逻辑、无指数退避、无断路器半开状态
- 401 和 503 同样处理（瞬态 vs 永久错误不区分）
- 同步 TCP 探测阻塞 tokio 运行时

**最终报告**：`REVIEW_REPORT.md`（1018 行，53 个问题）

## 2. 工具设计讨论：`line_edit`

### 问题
合并审查报告时，现有工具有缺口：
- `append_file` — 只能追加到末尾
- `patch_file` — 内容匹配而非位置匹配，易误伤
- `write_file` — 要求完整重写整个文件（33KB 一次性写入）

### 提出的新工具：`line_edit`
基于行号的编辑器工具，支持 6 种操作：
1. **insert_after** — 在某行后插入
2. **insert_before** — 在某行前插入
3. **replace_range** — 替换一段行范围
4. **delete_range** — 删除一段行范围
5. **replace_after_match** — 匹配后替换
6. **replace_before_match** — 匹配前替换

### 关键设计
- 相对行号：后续操作引用前序操作后的当前行号
- 原子性：所有操作一次提交
- dry_run 支持
- 最终方案写入 `PROPOSAL_line_edit.md`

## 3. 工具设计讨论：Agent 自创工具

### 用户问题
"是否可以引入一个工具台让 agent 可以自己创建工具"

### 分析
| 方案 | 安全风险 | 灵活性 |
|------|---------|--------|
| ❌ 完全自创工具（任意代码） | 🔴 灾难性（沙箱逃逸、无限递归） | ⭐⭐⭐⭐⭐ |
| ✅ 工具组合器（Tool Composer） | 🟢 低（仅组合现有工具） | ⭐⭐⭐⭐ |
| ✅ 工具别名 + 参数预制 | 🟢 低 | ⭐⭐⭐ |
| ✅ 条件路由器 | 🟡 中 | ⭐⭐⭐⭐ |

### 推荐路线
```
完全内建工具 → 可控组合工具 → 沙箱脚本 → 任意代码（不推荐）
（当前状态）   （推荐方案）   （受限）
```

### 核心原则
工具是**安全边界的一部分**。每个内建工具经过 schema 验证、输出截断、副作用控制、审计追踪。让 agent 创建任意工具相当于让 LLM 自己保证安全——而这正是审查报告里发现大量 `unsafe` 问题的同一个 LLM。

## 4. Agent 记忆架构讨论

### 用户核心观点（关键洞察）
**"定期检查可以让系统完成"**

### 当前提示词的问题
1. **只有"是什么"，没有"什么时候用"** — 没有触发条件、生命周期、同步策略
2. **没有元认知指令** — 没有要求 agent 在特定情况下主动读写 memo
3. **没有记忆层次** — 工作记忆、情景记忆、语义记忆混为一谈
4. **缺少子代理协作闭环** — 子代理不能写父 agent 的 memo

### 最佳分工（核心结论）
| 责任方 | 职责 |
|--------|------|
| **系统（自动化）** | 存储、TTL 过期、自动摘要、跨 session 注入、语义检索（RAG） |
| **Agent（人工判断）** | 内容选择、重要性标记 |

### 自动化层次
| 层次 | 描述 | 实现难度 |
|------|------|---------|
| L1 | write_memo 加 `ttl_seconds` 参数，到期自动删除 | 低 |
| L2 | 超阈值自动压缩旧 memo，或自动合并重复 memo | 中 |
| L3 | 跨 session 持久化 + 新 session 自动注入历史摘要 | 高 |
| L4 | 主动语义检索（类 RAG 架构） | 高 |

### 命名空间建议
- `task/{id}` — 当前任务发现，可自动清理
- `decision/*` — 架构/设计决策，跨 session 保留
- `context/summary` — 上下文摘要，每步自动更新
- `findings/{module}` — 长期保留的发现

### 证据：这次对话中 agent 的 memo 使用
- 整个对话只有 **3 次** memo 调用
- 用户问"是否差让 agent 主动记忆"后，agent 才意识到自己的 memo 使用不足
- **这本身就是提示词不够好的证明**

## 5. 跨主题主线

三个工具/架构讨论（line_edit、自创工具、记忆管理）共享一个核心问题：

> **什么能力应该内置在系统里，什么能力应该交给 agent？**

| 维度 | 应该系统做 | 应该 agent 做 |
|------|-----------|--------------|
| 安全 | 参数验证、沙箱、审计 | 选择哪个工具 |
| 资源 | 生命周期、清理、限流 | 分配合适的资源量 |
| 记忆 | 存储、检索、过期 | 选择值得记什么 |
| 执行 | 原子性、回滚、日志 | 决定执行顺序 |

**设计原则**：可靠的事交给系统（系统不会忘），需要判断的事交给 agent（agent 更有上下文）。
